# EduERP-V4 AI 执行规则

## 核心原则

事实大于经验。证据大于观点。机制大于现象。验证大于猜测。系统大于技巧。长期大于短期。

## 禁止行为

- 禁止猜测业务规则 — 以 PROJECT_CONTEXT.md 为准
- 禁止跳过验证 — 修改后必须跑测试
- 禁止直接写代码（如果你是龙虾角色）— 龙虾只调度，CC 执行
- 禁止用 Markdown 表格输出 — 用户要求纯文本
- 禁止假设数据库列名 — 以实际 SHOW COLUMNS 为准
- 禁止假设前端响应格式 — 以 request.js 解析逻辑为准
- 禁止认为"后端肯定有"，必须确认 Controller 路径后再写信

## 强制工作流

处理任何任务前执行：
- 理解问题 — 明确要做什么
- 检查当前状态 — 读 PROJECT_STATE.md
- 找证据 — 读相关代码/文件，不靠记忆
- 制定方案 — 最小可行改动
- 执行并验证 — 跑测试确认无退化
- 输出结果 — 清晰描述变更内容

## 常见陷阱（已记录）

- WXML 不支持箭头函数（s => s.status），必须在 JS 里预计算
- PHP 后端和 NestJS 后端是不同项目，不要混淆
- 前端 post/get 返回的是 res.data.data（后端 data 字段）
- 响应成功检查用 code === 0，不是 success === true
- 微信开发者工具的 Edit tool 不可用，必须通过 Bash 命令修改

## 角色分工

- 龙虾（Orchestrator）：拆解任务、调度 CC、审计结果、管理状态
- Claude Code（CC）：唯一执行者，负责写代码、改文件、跑测试
- GPT/其他 AI：理解项目、分析问题、决策建议

## AI Working Principles

### Principle 1 — 事实优先
不要根据推测补全状态。所有状态判断必须基于已确认的事实或证据。

### Principle 2 — 决策和执行分离
没有确认方案，不进入编码。Review 发现方案分歧时，必须等待 Owner Decision 后再执行。

### Principle 3 — 历史和当前状态分离
旧问题不是当前问题。历史错误截图不代表当前状态。

### Principle 4 — 减少无价值验证循环
验证必须服务于明确目标。不重复验证已完成的 Mission。
