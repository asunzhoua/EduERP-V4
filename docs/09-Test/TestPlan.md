# EduERP V4 — Test Plan

> 版本：v4.0
> 状态：架构冻结
> 最后更新：2026-07-06
> 文档编号：EDUOS-QA-001

---

## 一、设计目标

EduOS 所有模块开发完成以后。

必须先测试。

测试通过。

才能进入下一模块。

禁止：

多个模块一起开发。

最后一起测试。

---

## 二、测试原则

整个系统遵守以下原则：

**原则一**

一个功能。

对应一组测试。

**原则二**

开发完成。

立即测试。

**原则三**

发现问题。

立即修复。

不得累计Bug。

**原则四**

同一个Bug。

禁止出现第二次。

**原则五**

所有测试。

必须记录。

---

## 三、测试流程

统一流程：

```
开发

↓

自测

↓

修复

↓

再次测试

↓

通过

↓

提交
```

任何一步失败。

返回上一阶段。

---

## 四、测试分类

整个系统：

统一：

五类测试。

```
功能测试

权限测试

数据测试

异常测试

页面测试
```

整个流程：

全部成功。

才算通过。

## 二十二、Bug等级

统一三级。

**P0 — 系统无法使用**

例如：无法登录、无法签到、重复扣课、工资错误、数据库错误。

立即修复。

**P1 — 影响正常使用**

例如：数据显示错误、页面异常、消息未发送。

当天修复。

**P2 — 体验问题**

例如：颜色、文字、排版。

下一版本修复。

---

## 二十三、AI开发规范

AI完成一个模块。

必须执行：

```
① 功能测试
② 权限测试
③ 数据测试
④ 页面测试
⑤ Bug修复
⑥ 再测试
⑦ 输出测试报告
⑧ 等待下一任务
```

禁止：

连续开发多个模块。

---
