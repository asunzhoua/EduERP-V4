# EduOS 提醒机制设计方案

## 一、概述

### 1.1 目标
为教培场景提供及时、准确的通知提醒，提升用户体验和运营效率。

### 1.2 技术方案
- **应用内通知**：小程序内消息中心（兜底方案）
- **微信订阅消息**：一次性订阅，高价值场景使用

---

## 二、学生端提醒

### 2.1 上课提醒（P0）
**场景**：课程开始前 30 分钟提醒学生/家长
**触发条件**：
- 当前时间 = 课程开始时间 - 30 分钟
- 课程状态 = SCHEDULED
- 学生已报名该课程

**通知内容**：
```
【上课提醒】
课程：{课程名称}
时间：{日期} {开始时间}-{结束时间}
班级：{班级名称}
教师：{教师姓名}
```

**技术实现**：
- 后端定时任务（每 5 分钟扫描一次）
- 查询未来 30-35 分钟内的课程
- 发送微信订阅消息（需用户提前授权）
- 同时写入应用内消息中心

**需要决策**：
- [ ] 提醒时间：课前 30 分钟？还是可配置（15/30/60 分钟）？
- [ ] 是否需要家长也收到提醒？

---

### 2.2 缺勤提醒（P1）
**场景**：学生未到时，提醒家长关注
**触发条件**：
- 课程已结束
- 学生出勤状态 = ABSENT
- 教师已确认出勤

**通知内容**：
```
【缺勤提醒】
{学生姓名} 在 {日期} 的 {课程名称} 课程中缺勤。
如有疑问，请联系教师。
```

**技术实现**：
- 课程结束后 10 分钟触发
- 检查出勤状态
- 发送给学生/家长

**需要决策**：
- [ ] 是否发送？还是只在应用内显示？
- [ ] 迟到（LATE）是否也提醒？

---

### 2.3 学习进度提醒（P2）
**场景**：定期提醒学生查看学习进度
**触发条件**：
- 每周日晚上 20:00
- 学生本周有课程

**通知内容**：
```
【学习周报】
本周课程：{N} 节
出勤：{M} 节
缺勤：{K} 节
继续加油！
```

**技术实现**：
- 定时任务（每周日 20:00）
- 统计本周数据
- 发送给学生/家长

**需要决策**：
- [ ] 是否需要？频率如何（每周/每月）？

---

### 2.4 合同到期提醒（P1）
**场景**：合同即将到期，提醒续费
**触发条件**：
- 合同到期前 7 天
- 合同状态 = ACTIVE

**通知内容**：
```
【合同到期提醒】
您的 {课程名称} 合同将于 {到期日期} 到期。
剩余课时：{N} 节
请及时续费，继续学习！
```

**技术实现**：
- 定时任务（每天扫描）
- 查询到期前 7 天的合同
- 发送给学生/家长

**需要决策**：
- [ ] 提醒时间：到期前 7 天？3 天？1 天？

## 三、教师端提醒

### 3.1 待处理考勤提醒（P0）
**场景**：课程结束后，提醒教师确认出勤
**触发条件**：
- 课程已结束
- 出勤状态 = PENDING（未确认）
- 课程结束超过 1 小时

**通知内容**：
```
【待处理考勤】
您有 {N} 节课程的出勤未确认。
请及时处理。
```

**技术实现**：
- 定时任务（每小时扫描）
- 查询 PENDING 状态的出勤
- 发送给教师

**需要决策**：
- [ ] 是否发送？还是只在 Dashboard 显示？

---

### 3.2 即将上课提醒（P1）
**场景**：课程开始前 15 分钟提醒教师
**触发条件**：
- 当前时间 = 课程开始时间 - 15 分钟
- 课程状态 = SCHEDULED
- 教师 = 当前用户

**通知内容**：
```
【即将上课】
课程：{课程名称}
时间：{开始时间}-{结束时间}
班级：{班级名称}
学生：{N} 人
```

**技术实现**：
- 定时任务（每 5 分钟扫描）
- 查询未来 15-20 分钟内的课程
- 发送给教师

**需要决策**：
- [ ] 提醒时间：课前 15 分钟？还是可配置？

---

### 3.3 班级异常提醒（P2）
**场景**：学生连续缺勤或出勤率下降
**触发条件**：
- 学生连续 3 次缺勤
- 或学生出勤率 < 70%

**通知内容**：
```
【班级异常】
学生 {姓名} 在 {班级名称} 中：
连续缺勤 {N} 次
出勤率：{X}%
建议关注。
```

**技术实现**：
- 定时任务（每天扫描）
- 统计学生出勤数据
- 发送给教师

**需要决策**：
- [ ] 是否需要？阈值如何设定？

---

## 四、EOS 系统提醒

### 4.1 Mission 停滞提醒（P0）
**场景**：Mission 超过 30 分钟无进展
**触发条件**：
- Mission 状态 = RUNNING
- 最近 30 分钟无新 Commit/Evidence

**通知内容**：
```
【Mission 停滞】
Mission: {ID}
Phase: {Phase}
Batch: {Batch}
最后活动：{时间}
请检查 Executor 状态。
```

**技术实现**：
- Heartbeat 机制（每 30 分钟）
- 检查最近 Commit 时间
- 发送飞书通知

**需要决策**：
- [ ] 停滞时间阈值：30 分钟？还是可配置？

---

### 4.2 Decision 等待提醒（P1）
**场景**：Decision Gate 等待超过设定时间
**触发条件**：
- Mission 状态 = WAITING_DECISION
- 等待时间 > 1 小时

**通知内容**：
```
【Decision 等待】
Mission: {ID}
等待时间：{N} 小时
决策内容：{描述}
请及时处理。
```

**技术实现**：
- Heartbeat 机制
- 检查 Decision Gate 时间
- 发送飞书通知

**需要决策**：
- [ ] 等待时间阈值：1 小时？还是可配置？

---

### 4.3 Executor 异常提醒（P0）
**场景**：CC 执行超时或失败
**触发条件**：
- spawn_subagent 超时（> 10 分钟）
- 或返回错误

**通知内容**：
```
【Executor 异常】
Mission: {ID}
Batch: {Batch}
错误：{描述}
请检查 CC 状态。
```

**技术实现**：
- spawn_subagent 错误捕获
- 发送飞书通知

**需要决策**：
- [ ] 超时时间：10 分钟？还是可配置？

## 五、实施优先级

### Phase A（MVP，1-2 周）
1. 学生：上课提醒（P0）
2. 教师：待处理考勤提醒（P0）
3. EOS：Mission 停滞提醒（P0）

### Phase B（增强，2-4 周）
4. 学生：缺勤提醒（P1）
5. 学生：合同到期提醒（P1）
6. 教师：即将上课提醒（P1）
7. EOS：Decision 等待提醒（P1）

### Phase C（完善，4-8 周）
8. 学生：学习进度提醒（P2）
9. 教师：班级异常提醒（P2）
10. EOS：Executor 异常提醒（P0）

## 六、技术架构

reminder/
├── reminder.entity.ts
├── reminder.service.ts
├── reminder.controller.ts
├── reminder-scheduler.service.ts（定时任务）
└── wechat-message.service.ts（微信订阅消息）

### 6.2 数据库
```sql
CREATE TABLE reminder (
  id INT PRIMARY KEY AUTO_INCREMENT,
  type VARCHAR(50), -- LESSON_REMIND, ABSENT_REMIND, etc.
  userId INT,
  title VARCHAR(200),
  content TEXT,
  status VARCHAR(20), -- PENDING, SENT, READ
  createdAt DATETIME,
  sentAt DATETIME
);
```

## 八、实施建议

### 建议 1：分阶段实施
- 先做 P0（核心提醒）
- 验证效果后再做 P1/P2

### 建议 2：应用内通知优先
- 先实现应用内消息中心
- 微信订阅消息作为增强

### 建议 3：可配置化
- 提醒时间、阈值等参数可配置
- 避免硬编码

### 建议 4：用户授权
- 微信订阅消息需要用户授权
- 在首页/设置页添加"订阅提醒"按钮
