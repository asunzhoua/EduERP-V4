# EduERP V4 — Deploy Guide

> 版本：v4.0
> 状态：架构冻结
> 最后更新：2026-07-06
> 文档编号：EDUOS-OPS-001

---

## 一、设计目标

保证系统：

稳定运行。

数据安全。

快速恢复。

降低维护成本。

整个系统：

宁可简单。

不要复杂。

---

## 二、运行原则

整个系统遵守以下原则。

**原则一：稳定第一**

**原则二：数据第一**

**原则三：先备份，后修改**

**原则四：任何修改，可以恢复**

**原则五：系统出现异常，优先恢复，不要继续开发**

---

## 三、服务器

整个系统统一一套服务器。

运行：

```
API服务
数据库
EventBus
消息服务
```

禁止多套业务系统。

---

## 四、数据库

数据库统一MySQL。

所有业务统一一个主数据库。

禁止多个数据库保存唯一数据。

---

## 五、图片存储

所有图片统一保存，包括：

```
学生头像
课程反馈图片
请假证明
积分商品图片
```

图片不得存入数据库。

---

## 六、日志

所有日志统一保存，包括：

```
系统日志
错误日志
操作日志
接口日志
```

日志禁止删除。

---

## 七、自动备份

每天自动备份。

时间建议凌晨：

```
02:00 自动备份数据库
```

无需人工操作。

---

## 八、备份内容

每天统一备份：

```
数据库
上传图片
系统配置
日志
```

全部保存。

---

## 九、备份保存周期

建议保留：

```
最近7天
最近30天
最近90天
```

超过时间自动删除。

---

## 十、恢复原则

如果出现以下情况，立即恢复：

- 服务器异常
- 数据库异常
- 误操作

禁止人工修改数据库。

---

## 十一、恢复流程

统一流程：

```
停止系统
    ↓
恢复数据库
    ↓
恢复图片
    ↓
恢复配置
    ↓
启动系统
    ↓
检查数据
    ↓
开放使用
```

全部完成才能恢复服务。

---

## 十二、升级原则

升级必须提前备份。

升级失败立即恢复。

禁止边升级边使用。

---

## 十三、修改规则

修改任何规则（工资规则、课时规则、课程价格、积分规则），必须记录：

- 修改时间
- 修改人员
- 修改原因

---

## 十四、服务器监控

每天检查：

```
CPU
内存
磁盘
数据库
接口
```

发现异常立即处理。

---

## 十五、数据库监控

检查：

```
连接数
运行状态
备份状态
剩余空间
```

数据库保持正常。

---

## 十六、接口监控

重点检查：

```
登录
签到
扣课
工资
消息
请假
```

出现异常立即处理。

---

## 十七、异常处理

统一流程：

```
发现问题
    ↓
记录日志
    ↓
定位原因
    ↓
恢复系统
    ↓
再次测试
    ↓
恢复正常
```

禁止直接修改数据。

---

## 十八、数据修改

任何数据（课时、工资、积分、收费）修改只能通过后台。

禁止直接修改数据库。

---

## 十九、权限

服务器只有管理员、老板拥有最高权限。

普通老师、家长无法进入服务器。

---

## 二十、密码

所有后台账号必须修改默认密码。

密码不得公开、不得共享。

---

## 二十一、系统关闭

服务器维护时统一关闭系统。

维护完成重新开启。

禁止部分功能关闭。

---

## 二十二、恢复测试

每个月至少恢复一次备份。

确认备份可用、恢复成功。

---

## 二十三、上线检查

上线前统一检查：

```
数据库正常
接口正常
图片正常
消息正常
备份正常
日志正常
权限正常
```

全部通过允许上线。

---

## 二十四、AI开发规范

AI不得：

- 修改服务器配置
- 删除数据库
- 直接执行危险命令

所有修改必须输出方案，等待Architect确认才能执行。

---

## 二十五、禁止事项

- 禁止直接删除数据库
- 禁止直接删除学生
- 禁止直接删除老师
- 禁止修改数据库数据
- 禁止关闭日志
- 禁止跳过备份

---

## 二十六、上线标准

EduOS V1.0 满足以下条件允许正式上线：

```
服务器正常
数据库正常
接口正常
消息正常
日志正常
备份正常
恢复正常
测试通过
```
