# ADR-014: Event Strategy

> **Status**: PROPOSED
> **Date**: 2026-07-15
> **Sprint**: 5 WP1
> **Supersedes**: (none ÔÇö formalizes existing event architecture)
> **Relates to**: Constitution Rule 21 (Event Publishing), EventCatalog.md, EventSchema.md

---

## Context

EduOS uses `@nestjs/event-emitter` (EventEmitter2) for inter-context communication. The event system is the backbone of the architecture (Constitution Rule 2: Event First). Without formal event strategy rules, developers may:
1. Emit events that violate naming conventions
2. Create events without proper schemas
3. Forget idempotency keys
4. Emit events from the wrong context
5. Create circular event chains

---

## Decision

**Adopt the following event strategy rules:**

### Rule ES-001: Event Naming Convention

All events follow the pattern: `<domain>.<action>[.<context>]`

| Component | Rule | Example |
|-----------|------|---------|
| Domain | Lowercase, matches publisher | `lesson`, `contract`, `student` |
| Action | Past tense verb | `completed`, `finished`, `exhausted` |
| Context | Optional, lowercase | `feedback.created`, `status.changed` |

**Examples:**
- `lesson.completed` (Teaching domain, lesson action)
- `contract.exhausted` (Finance domain, contract action)
- `student.status.changed` (Student domain, student action with context)

### Rule ES-002: Event Schema Standard

Every event payload MUST include these envelope fields:

```typescript
interface EventEnvelope {
  eventId: string;      // UUID v4, generated by EventBusService
  timestamp: string;    // ISO-8601, generated by EventBusService
  version: string;      // Semver (e.g., '1.0.0')
}
```

Plus domain-specific fields. See EventSchema.md for complete schemas.

### Rule ES-003: Idempotency Key

Every event MUST declare an idempotency key. Consumers MUST use this key to prevent duplicate processing.

| Event | Idempotency Key | Reason |
|-------|----------------|--------|
| `lesson.completed` | `lessonId` | Same lesson, same completion |
| `lesson.finished` | `lessonId` | Same lesson, same financial settlement |
| `attendance.confirmed` | `lessonId` | Same lesson, same confirmation |
| `contract.exhausted` | `contractId` | Same contract, same exhaustion |
| `student.deactivated` | `studentId` | Same student, same deactivation |

### Rule ES-004: One Publisher Per Event

Each event has exactly one publisher context. No two contexts emit the same event name.

| Event | Publisher | Enforced By |
|-------|-----------|-------------|
| `lesson.completed` | Teaching | Module boundary |
| `lesson.finished` | Teaching | Module boundary |
| `contract.exhausted` | Finance | Module boundary |
| `student.deactivated` | Student | Module boundary |

### Rule ES-005: Event Ordering Guarantee

Certain events MUST be emitted in order. Consumers may depend on this ordering.

**Required Order (Teaching):**
1. `lesson.completed` (TEACHING Ôćĺ FINISHED)
2. `attendance.confirmed` (all attendance confirmed)
3. `lesson.finished` (FINISHED Ôćĺ ARCHIVED)

**Constraint:** `lesson.finished` MUST NOT be emitted before `lesson.completed` and `attendance.confirmed`.

### Rule ES-006: No Circular Event Chains

Events must not create circular chains. If Event A triggers Context X to emit Event B, Event B must not trigger Context Y to emit Event A.

**Allowed:**
- `lesson.finished` Ôćĺ Finance Ôćĺ `contract.exhausted` Ôćĺ Teaching (different event, different purpose)

**Prohibited:**
- `lesson.finished` Ôćĺ Finance Ôćĺ `points.granted` Ôćĺ Points Ôćĺ `lesson.finished` (circular)

### Rule ES-007: Event Payload Immutability

Event payloads are immutable for the lifetime of the major version. Adding new fields requires a minor version bump. Removing or changing fields requires a major version bump.

### Rule ES-008: Event Registration Before Implementation

New events MUST be registered in EventCatalog.md and EventSchema.md before implementation. No event may be emitted without a catalog entry.

## Event Processing Model

### Synchronous Processing (Current)

`@nestjs/event-emitter` processes events synchronously within a single NestJS process.

**Advantages:**
- Strong ordering guarantees
- Transactional consistency within a process
- Simple error handling

**Disadvantages:**
- Single process bottleneck
- No horizontal scaling for event processing
- Event processing failure blocks the emitter

### Future: Asynchronous Processing

When scaling requires it, events may be moved to a message queue (RabbitMQ, Kafka).

**Requirements for migration:**
- Event ordering must be preserved
- Idempotency must be maintained
- Dead letter queue for failed events
- Retry mechanism with exponential backoff

## Event Versioning Strategy

| Change Type | Version Bump | Example |
|------------|-------------|---------|
| Adding optional field | Patch (1.0.x) | Add `note` field to `lesson.finished` |
| Adding required field with default | Minor (1.x.0) | Add `confirmedBy` with default 0 |
| Removing field | Major (x.0.0) | Remove `durationMinutes` from `lesson.completed` |
| Changing field type | Major (x.0.0) | Change `lessonId` from number to string |

---

## Consequences

### Positive
- Clear, enforceable event strategy rules
- Prevents common event-related bugs
- Aligns with Constitution Rule 2 (Event First) and Rule 21 (Event Publishing)
- Provides migration path to asynchronous processing

### Negative
- Requires discipline to maintain event catalog
- Synchronous processing may become a bottleneck at scale

---

*This ADR defines the event strategy. All developers must comply. Events are the primary communication mechanism in EduOS.*
