## 2. 详细审计

### 2.1 状态变化检测审计

| 触发 | Defined | 通知级别 | 是否有去重 |
|:-----|:--------|:---------|:----------|
| CREATED → RUNNING | ✅ | P2 | ✅ 同一变化不重复 |
| RUNNING → COMPLETED | ✅ | P1 | ✅ |
| RUNNING → FAILED | ✅ | P0 | ✅ |
| RUNNING → PAUSED | ✅ | P0 | ✅ |

**未覆盖的场景**：PAUSED → RUNNING（resume）。没有单独定义这个通知，因为 resume 会伴随状态变化，通知 RUNNING 即可。

**结论**：✅ PASS — 覆盖所有关键状态变化

### 2.2 异常检测审计

| 检测对象 | 场景 | 是否定义 | Severity |
|:---------|:-----|:---------|:---------|
| Pump Runner | 连续失败 ≥ 2 | ✅ | P0 |
| Pump Runner | CRASH | ✅ | P0 |
| Runtime | RUNNING 超时 > 30min | ✅ | P1 |
| Runtime | Evidence 4h 未更新 | ✅ | P1 |
| Runtime | state 损坏 | ✅ | P1 |
| Feishu | API 调用失败 ≥ 3 | ✅ | P1 |
| Feishu | Token 异常 | ✅ | P0 |
| Feishu | Board 无响应 | ✅ | P1 |

**未覆盖的场景**：Claude Code CLI 自身异常（如 OAuth token 过期）。这属于 Pump Runner 执行时的问题，Pump Runner 的 exit code 和 stderr 会捕获。

**结论**：✅ PASS — 8 种异常检测覆盖充分

### 2.3 渠道方案审计

| 维度 | 方案 A: Webhook | 方案 B: App 消息 | 方案 C: 事件订阅 |
|:-----|:----------------|:-----------------|:-----------------|
| 实现成本 | 低 | 中 | 高 |
| 可靠性 | 中 | 高 | 中 |
| 个人通知 | ❌ | ✅ | — |
| 操作按钮 | ✅ 卡片 | ✅ 卡片 | — |
| 当前基础设施 | 未配置 | API 凭证已有 | 无 |

**推荐**：方案 A（主通道）+ 方案 B（P0 通道）

**理由**：
- 方案 A: 30 分钟配置，飞书群内可见，多人可查
- 方案 B: P0 需要更高可靠性 + 个人直达 + 操作按钮
- 方案 C: 不选 — 需要事件服务器，维护成本高，当前收益低

**结论**：✅ PASS — 方案选择有充分依据

### 2.4 边界审计

```
通知系统的操作边界：
                        ┌──────────────────┐
                        │  通知系统         │
                        │                   │
                        │  ✅ 发送消息      │
                        │  ✅ 读取 Mission   │
                        │  ✅ 读取 Runtime   │
                        │  ❌ 修改 Mission   │
                        │  ❌ 修改 Runtime   │
                        │  ❌ 调用 CC       │
                        │  ❌ 启动 Runner   │
                        └──────────────────┘
```

**边界检查对比（与现有架构原则）：**

| 架构原则 | 通知系统是否违反 | 理由 |
|:---------|:----------------|:------|
| 飞书不是 Runtime | ❌ 不违反 | 通知只读飞书，不写 |
| 龙虾不是 Executor | ❌ 不违反 | 通知是消息发送，不是代码执行 |
| Pump Runner 唯一调度层 | ❌ 不违反 | 通知不触发调度 |
| CC 唯一执行者 | ❌ 不违反 | 通知不调用 CC |
| 自动发现 ≠ 自动修改 | ❌ 不违反 | 只发现，不修改 |
| 自动提醒 ≠ 自动决策 | ❌ 不违反 | 只提醒，决策权在主人 |

**结论**：✅ PASS — 所有架构边界保持

## 3. 风险评估

| 风险 | 概率 | 影响 | 缓解措施 |
|:-----|:-----|:-----|:---------|
| 通知风暴（大量状态变化导致刷屏） | 低 | 中 | P0 即时 / P1 每分钟合并 / P2 每次 Heartbeat |
| 通知丢失（Webhook 不可达） | 中 | 低 | 通知失败不影响系统运行；P0 可回退到方案 B |
| 误报（健康系统被报告异常） | 低 | 中 | 阈值加安全边际 + 状态变化去重 |
| 延迟通知 | 低 | 低 | Heartbeat 15-30 分钟周期是设计选择 |

**残存风险**：无不可接受的风险。
