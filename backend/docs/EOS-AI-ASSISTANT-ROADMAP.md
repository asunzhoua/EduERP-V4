## 3. Phase 0：前置条件

**优先级**: P0 — 必须完成  
**工期**: 1-3天  
**风险**: 🟢 低（但审批时间不确定）

### 3.1 任务清单

| 序号 | 任务 | 描述 | 预估工时 | 依赖 |
|:-----|:-----|:------|:---------|:-----|
| 0.1 | 开通 `im:message` 权限 | 在飞书开发者后台 → 应用 → EOS → 权限管理 → 勾选 `im:message` | 15分钟 | 开发者账号 |
| 0.2 | 开通 `im:resource` 权限（可选） | 如需发送图片/文件，勾选对应权限 | 5分钟 | 0.1 |
| 0.3 | 配置 Event Subscription | 设置事件回调 URL（需事先准备服务器地址） | 15分钟 | 0.1 |
| 0.4 | 配置事件类型 | 勾选 `im.message.receive_v1` | 5分钟 | 0.1 |
| 0.5 | 重新发布应用 | 飞书开发者后台 → 版本管理 → 创建新版本 → 提交发布 | 15分钟 | 0.1-0.4 |
| 0.6 | **等待审批** | 飞书审核应用发布（通常1-3个工作日） | **1-3天** | 0.5 |

### 3.2 审批加速建议

- 飞书自建应用的审批通常由租户管理员（即主人本人）审批，无需等待外部
- 如果主人是管理员，审批流程为：提交发布 → 在管理后台 → 审批管理 → 通过
- 审批通过后，应用权限立即生效

### 3.3 风险

| 风险 | 概率 | 影响 | 缓解 |
|:-----|:------|:------|:------|
| 审批被拒 | 低 | 阻塞所有后续阶段 | 确认权限选择正确 |
| 审批人不在 | 低 | 等待1-2天 | 提前联系审批人 |
| 权限理解偏差 | 低 | 权限不足 | 开发前核对权限列表 |

## 4. Phase 1：消息通道

**优先级**: P0 — 必须完成  
**工期**: 2-4天  
**风险**: 🟡 中（涉及新代码）

### 4.1 架构

```
飞书 Event Subscription
        │
        ▼ (POST JSON + 签名)
┌─────────────────────────────────────┐
│    Event 处理服务器（轻量）           │
│                                      │
│  1. 验证签名（X-Lark-Signature）      │
│  2. 解析事件类型                      │
│  3. 提取消息内容 + 发送者信息          │
│  4. 验证是否 @机器人 提及             │
│  5. 过滤非 @消息                     │
│  6. 传给 Layer 1（Lobster）          │
│  7. 将 Layer 1 回复发送回飞书 API     │
└─────────────────────────────────────┘
```

### 4.2 任务清单

| 序号 | 任务 | 描述 | 预估工时 |
|:-----|:-----|:------|:---------|
| 1.1 | 创建事件处理服务器 | 简单的 HTTP Server（Python/Flask 或 Node/Express） | 4h |
| 1.2 | 验证 Event Subscription | 飞书要求配置回调 URL 后发送挑战验证（Challenge Check） | 1h |
| 1.3 | 实现签名验证 | 验证 X-Lark-Signature | 1h |
| 1.4 | 解析消息事件 | 处理 `im.message.receive_v1` payload | 2h |
| 1.5 | 过滤 @提及 | 只处理含有 @EOS AI Assistant 的消息 | 1h |
| 1.6 | 实现消息发送 | 调用飞书 API `im/v1/messages` 发送回复 | 2h |
| 1.7 | 端到端测试 | 在飞书群 @机器人 → 收到回复 | 2h |

### 4.3 关键设计决策

| 决策 | 选项 | 推荐 | 理由 |
|:-----|:-----|:-----|:------|
| 服务器语言 | Python / Node | **Python** | 与现有工具链一致 |
| 服务器位置 | 本地 / 云函数 | **本地** | 减少基础设施，Lobster 即服务器 |
| 消息传递方式 | 内部函数调用 / HTTP | **内部函数** | 减少延迟，不出本地进程 |
| 持久化存储 | 需要 / 不需要 | **不需要** | 无状态设计，简化运维 |

### 4.4 相关飞书 API

| API | 用途 | 文档 |
|:----|:-----|:------|
| `POST /open-apis/im/v1/messages` | 发送消息 | 飞书开放平台 |
| `POST /open-apis/im/v1/messages/{message_id}/reply` | 回复消息 | 飞书开放平台 |
| `im.message.receive_v1` | 事件订阅类型 | 飞书事件列表 |

## 5. Phase 2：命令处理

**优先级**: P1 — 核心功能  
**工期**: 3-5天  
**风险**: 🟡 中（NLP 解析不确定性）

### 5.1 任务清单

| 序号 | 任务 | 描述 | 预估工时 |
|:-----|:-----|:------|:---------|
| 2.1 | 实现命令解析器 | NLP 自由输入 → 结构化为已知命令（参见架构设计命令格式） | 4h |
| 2.2 | 实现查询接口 | status / list_missions / show_mission / show_evidence / health | 4h |
| 2.3 | 实现控制接口 | create_mission / resume_mission / abort_mission / cancel_mission | 4h |
| 2.4 | 实现权限验证 | 白名单 + 前置条件检查 + 确认流程 | 2h |
| 2.5 | 实现确认流程 | 需要确认的操作 → 等待确认 → 执行/取消 | 2h |
| 2.6 | 格式化输出 | 返回结构化回复文本 | 2h |
| 2.7 | 集成测试 | 全命令覆盖测试 | 3h |

### 5.2 命令解析逻辑

```
原始命令: "@EOS AI Assistant 查看任务 M-001 的详细状态"

步骤1：去前缀 → "查看任务 M-001 的详细状态"
步骤2：NLP 关键词提取
  └─ 主命令: "查看任务" / "mission" (show_mission)
  └─ 参数: "M-001" (mission_id)
  └─ 修饰: "详细状态" (展示级别)
步骤3：匹配模式
  └─ show_mission(mission_id="M-001", detail="详细")
步骤4：权限验证 → ✅ show_mission 在白名单
步骤5：执行 → 读取 .missions/M-001/state → 格式化输出

如果 NLP 无法理解：
  回退到关键词匹配（正则表达式）
  如果仍不匹配：返回"无法识别以下命令：..."
```

### 5.3 接口实现细节

每个接口的实现需要：

| 接口 | 读取源 | 写入目标 | 副作用 |
|:-----|:--------|:---------|:-------|
| `status` | `.missions/*/state` + `FEISHU_BOARD` | 无 | 无 |
| `list_missions` | `.missions/*/state` | 无 | 无 |
| `show_mission` | `.missions/{id}/state` + `evidence/` | 无 | 无 |
| `create_mission` | 飞书 Board（schema） | 飞书 Board 新增行 | 创建飞书记录 |
| `resume_mission` | `.missions/{id}/state` | 飞书 Board Status | 更新飞书 |
| `abort_mission` | `.missions/{id}/state` | 飞书 Board Status | 更新飞书 |
| `cancel_mission` | `.missions/{id}/state` | 飞书 Board Status | 更新飞书 |

所有接口最终回复通过 Layer 0 发送回飞书。

## 6. Phase 3：自动化链路

**优先级**: P1 — 闭环价值  
**工期**: 2-3天  
**风险**: 🟢 低（基于已验证组件）

### 6.1 任务清单

| 序号 | 任务 | 描述 | 预估工时 |
|:-----|:-----|:------|:---------|
| 3.1 | 飞书 Board → mission.json | 创建 Mission 后自动导出为 Pump Runner 可读格式 | 2h |
| 3.2 | Pump Runner 检测新任务 | 定时扫描飞书 Board 的 CREATED 任务 | 3h |
| 3.3 | 状态回写自动化 | Mission COMPLETED/FAILED/PAUSED 后更新飞书 Board | 2h |
| 3.4 | 全链路测试 | 飞书 @机器人 创建 → 自动执行 → 群通知 | 3h |
| 3.5 | 错误处理测试 | 各种异常路径 | 2h |

### 6.2 自动化流程

```
主人: @EOS AI Assistant 创建任务 优化数据库索引
  → Layer 0: 接收消息
  → Layer 1: create_mission("优化数据库索引")
    → 飞书 Board 新增行 (Status: CREATED)
    → 回复主人: "已创建任务 M-XXXX"
  → Layer 0: 发送回复到飞书

  → Pump Runner 定时扫描飞书 Board
    → 发现 M-XXXX (Status: CREATED)
    → 导出为 mission.json
    → 执行任务
    → 更新 .missions/M-XXXX/state → COMPLETED
    → 收集 Evidence

  → Layer 1 (Heartbeat) 检测到 COMPLETED
    → 审计 Evidence
    → 更新飞书 Board Status → COMPLETED
    → Webhook → 群通知: "M-XXXX 已完成 ✅"
    → 应用机器人 → 私信主人: "任务执行结果摘要"
```

### 6.3 Mission Board → mission.json 映射

| 飞书列 | mission.json 字段 | 映射方向 |
|:-------|:------------------|:---------|
| Mission ID | id | 双向 |
| Mission Name | name | 双向 |
| Description | description | 双向 |
| Status | status | 飞书→mission（仅 CREATED） |
| Priority | priority | 双向 |
| Owner | owner | 双向 |
| Executor | executor | 双向 |
| Started Time | started_at | mission→飞书 |
| Finished Time | finished_at | mission→飞书 |
| Evidence Link | evidence_url | mission→飞书 |

### 6.4 约束（与 Phase 3/4 Roadmap 一致）

- Pump Runner 只读取 CREATED 状态的 Mission
- Pump Runner 不修改飞书 Board，只写 Runtime `.missions/`
- 状态回写由 Lobster（Phase 1）审计后执行
- 飞书 Board 的状态验证规则需同步更新

## 7. Phase 4：增强功能

**优先级**: P2 — 非阻塞  
**工期**: 3-5天  
**风险**: 🟢 低  

### 7.1 可选功能清单

| 功能 | 描述 | 工時 | 优先级 |
|:-----|:------|:-----|:-------|
| 4.1 多轮对话 | 上下文保持，连续问问题 | 3h | P2 |
| 4.2 主动通知 | 任务完成/失败时私信主人 | 2h | P2 |
| 4.3 仪表盘集成 | 回复中附带格式化卡片 | 4h | P2 |
| 4.4 定时汇报 | 每日/每周自动推送摘要 | 3h | P3 |
| 4.5 权限分级 | 支持多人使用（Owner/Viewer） | 4h | P3 |
| 4.6 Webhook 管理 | 通过对话管理 Webhook | 2h | P3 |

### 7.2 多轮对话设计要点

- 使用消息 thread 关联上下文（飞书 `root_id`）
- 每对话上下文最多保持 10 轮
- 超过 10 轮或 30 分钟无消息 → 自动清除上下文
- 上下文存储：内存（无持久化）

### 7.3 增强功能依赖

| 功能 | 依赖 | 前提条件 |
|:-----|:------|:---------|
| 多轮对话 | Phase 2 | 命令解析已完成 |
| 主动通知 | Phase 1 + 现有 Webhook | 消息通道已完成 |
| 仪表盘 | 飞书 Card 消息 | `im:message` 权限支持富文本 |
| 定时汇报 | Phase 3 | 自动化链路已完成 |
| 权限分级 | 飞书用户认证 | Layer 0 用户信息提取 |

## 8. 风险与缓解

### 8.1 风险矩阵

| 编号 | 风险 | 概率 | 影响 | 阶段 | 缓解措施 |
|:-----|:------|:-----|:------|:-----|:---------|
| R1 | 飞书应用审批被拒 | 低 | 高（阻塞 P0-P4） | P0 | 提前核对权限，自建应用通常由自己审批 |
| R2 | Event Subscription 服务器不稳定 | 中 | 中（消息丢失） | P1 | 重试机制 + 死信队列 |
| R3 | NLP 命令解析不准 | 中 | 中（用户体验差） | P2 | 关键词回退 + 明确拒绝 + 可配置命令模板 |
| R4 | Pump Runner 与飞书 Board 状态冲突 | 低 | 高（数据不一致） | P3 | System of Record 约束，.missions/ 优先 |
| R5 | 飞书 API 限流 | 低 | 低（响应延迟） | P1 | 本地速率限制 + 重试 |
| R6 | GPT 输出幻觉 | 中 | 中（误导用户） | P2 | 状态来源于实际文件，GPT 仅格式化 |
| R7 | 审批流程过长 | 中 | 中（进度延迟） | P0 | 建议主人同时是审批人，当天可完成 |

### 8.2 风险应对策略

| 应对类型 | 策略 | 适用风险 |
|:---------|:------|:---------|
| 避免 | 权限最小化，只开通必要权限 | R1 |
| 减轻 | 事件服务器内建重试+日志 | R2 |
| 减轻 | 命令解析双引擎（NLP + 关键词） | R3 |
| 转移 | System of Record 明确冲突时以 Runtime 为准 | R4 |
| 接受 | API 限流影响不大，可接受短暂延迟 | R5 |
| 减轻 | 状态数据附加来源引用 | R6 |
