# EOS 项目执行规则 v1.0

> 生效日期：2026-07-19
> 状态：CONFIRMED
> 来源：用户指令，作为后续所有 Mission 执行的硬性约束

---

## 核心要求

后续所有 Mission 执行与验收，严格遵循：**以实际执行结果作为唯一判断依据，减少未经验证的推理和扩展。**

---

## 执行原则

### 1. 事实优先

任务推进必须基于：

- 实际代码状态
- 实际配置文件
- 实际运行日志
- 实际测试结果
- 实际数据库状态
- 实际执行输出

**禁止**根据经验或可能性提前判定问题。

### 2. 明确区分状态

所有问题必须分类：

| 状态 | 定义 |
|:-----|:------|
| **CONFIRMED** | 已通过实际证据确认 |
| **FIXED** | 已完成修复并验证 |
| **PENDING** | 已发现但未处理 |
| **UNKNOWN** | 未验证，不做判断 |

**禁止**将可能问题、理论风险、经验推测直接记录为项目问题。

### 3. Mission 边界控制

Mission 验收只关注：

- Mission 目标是否完成
- 相关问题是否解决
- 实际结果是否符合要求

**禁止**因为发现其他非阻塞问题：

- 自动扩大任务范围
- 增加额外优化
- 修改架构
- 创建无必要文档

### 4. 验收规则

完成判断必须提供证据，格式：

```
执行命令:
xxx
实际输出:
xxx
结果:
PASS / FAIL
```

**禁止**使用：应该可以、理论上没问题、推测已经完成。

### 5. 问题处理原则

**第一步**：确认是否影响当前目标。

**第二步**：
- 如果影响 → 创建修复任务、提供证据、修复验证
- 如果不影响 → 记录、等待后续 Mission

## 角色职责（保持不变）

| 角色 | 职责 |
|:-----|:------|
| **Lobster / Orchestrator** | 任务规划、Mission 调度、验收审核、状态管理 |
| **Claude Code** | 执行具体代码修改、输出执行证据、完成验证 |

**禁止角色混淆。**

---

## 总结

```
事实 > 推测
证据 > 判断
完成目标 > 扩展优化
实际结果 > 理论分析
```

所有 Mission 按此标准执行。
