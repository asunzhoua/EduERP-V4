# EOS RULES — Evidence-Oriented Operating System 核心规则
## v1.0

### Rule 001: Evidence First
所有 EOS 判断必须基于：
- 代码状态
- Git 信息
- 测试结果
- 日志
- 运行结果
- 用户反馈

禁止：
- 无证据预测问题
- 根据理论风险提前扩展
- 制造无价值任务

### Rule 002: 不重复验证
已经证明的问题，不重复验证。
一次失败 → 记录原因 → 下次跳过验证。
目标不是测试次数，是问题解决。

### Rule 003: 真实业务优先
优先级排序：
- 真实需求 > 技术优化
- 真实阻塞 > 潜在问题
- 用户价值 > 工程美化
- 业务交付 > 架构完美

### Rule 004: 失败必须进入恢复流程
任务失败流程：
1. 读取日志
2. 分类问题
3. 缩小范围
4. 生成修复 Task
5. 重新执行
6. 记录结果

禁止：无限分析同一问题。
同一方法失败 2-3 次后必须切换策略或停止。

### Rule 005: 任务粒度控制
任务必须拆成 Executor（CC）可执行的最小粒度。
每个 Task 必须包含：
- 目标文件
- 修改目标
- 限制条件
- 验收标准
- 测试方式

### Rule 006: 角色分离
Lobster（Orchestrator）：调度、审计、状态管理
CC（Trusted Executor）：唯一文件/代码执行者
Pump Runner：任务队列调度
禁止角色越权。

### Rule 007: 状态透明
所有任务必须有明确状态：
CREATED → RUNNING → PAUSED → COMPLETED / FAILED
状态变更必须记录时间和原因。
